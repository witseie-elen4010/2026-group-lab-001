# ADR 003: Adopt Jest for Automated Testing with GitHub Actions CI

## Context

ADR 001 established Express.js as the backend framework and identified Jest as the target
test runner for unit and service testing. ADR 002 introduced MongoDB Atlas as the
persistent database, which added a need for integration tests that can exercise real
collection queries against seeded Atlas data while remaining skippable in environments
where a live connection is unavailable.

As the project grows across sprints, the testing strategy must address three distinct
levels of verification:

- **Unit testing** of pure business logic — validation rules, utility functions, and data
  transformations that have no external dependencies and can run in isolation
- **Integration testing** of Express routes and model helpers — verifying that HTTP
  handlers produce correct responses, enforce authentication and ownership rules, and
  interact with MongoDB collections in the expected way, without requiring a live database
  connection for every run
- **System and acceptance testing** against a real Atlas instance — confirming that
  seeded reference data is present, that relationship lookups across University, Faculty,
  and School collections resolve correctly, and that the application behaves correctly
  end-to-end under realistic conditions

A CI pipeline is needed to enforce these tests automatically on every pull request to
`main`, so that regressions are caught before merging rather than discovered in
production. The project already uses Node.js 20 and npm, so the chosen tools must
integrate cleanly with the existing stack without requiring a separate test runner process
or build step.

## Decision

We have decided to use **Jest** as the unified test runner, assertion library, and mock
framework for all three testing tiers, and **GitHub Actions** to execute the full test
suite on pull requests and pushes to `main`.

The testing strategy is structured as follows:

- **Test Runner:** Jest 30, configured via `tests/jest.config.js` with
  `testEnvironment: node`, a `testMatch` glob covering all `*.test.js` files under
  `tests/`, and a `testTimeout` of 30 000 ms to accommodate route tests that spin up
  real HTTP servers
- **Unit Tests** (`tests/services/`, `tests/utils/`): Pure function tests with no
  external dependencies and no mocking. These cover business logic such as consultation
  preference validation rules (`consultation_preferences_validation.test.js`), institution
  selection validation (`institution_validation.test.js`), and calendar cell construction
  (`calendar.test.js`). They run entirely in-process and require no environment variables.
- **Integration Tests** (`tests/routes/`, `tests/models/`): Route handlers are tested
  by binding the Express application to a real `http.createServer` instance and issuing
  HTTP requests using Node's native `fetch`. MongoDB collections are replaced with
  `jest.mock` stubs so that tests can control return values and verify call arguments
  without connecting to Atlas. Session cookies are extracted from login responses and
  forwarded on subsequent requests to simulate authenticated sessions. Model helpers are
  tested in isolation by mocking the shared `getCollection` helper.
- **System and Acceptance Tests** (`tests/models/atlas.test.js`): These tests connect
  to the live Atlas instance using the `MONGODB_URI` environment variable. Each test is
  gated with a `test.skip` fallback so that the suite passes in environments where the
  secret is absent. They assert that the `LetsTalk` database is reachable, that the
  expected collections exist, that seeded institution records are present and correctly
  linked, and that relationship validation helpers return accurate results against real
  data.
- **Mocking Strategy:** `jest.mock` is used at the module level to replace database
  helpers, password utilities, and other I/O-bound dependencies in route and model tests.
  Unit tests use no mocks. Atlas tests use no mocks.
- **Coverage:** Coverage collection is enabled globally via `collectCoverage: true` in
  `package.json`. Reports are generated automatically on every test run and uploaded as
  a build artifact in CI.
- **Serial Execution:** The CI pipeline runs Jest with `--runInBand` to prevent port
  conflicts when multiple route test files bind a real HTTP server on the same port in
  the same process.

The GitHub Actions CI workflow (`.github/workflows/ci.yml`) is configured as follows:

- **Trigger:** Runs on pull requests targeting `main` and on direct pushes to `main`,
  with an additional manual `workflow_dispatch` trigger
- **Environment:** Ubuntu latest, Node.js 20, dependencies restored with `npm ci`
- **Steps:**
  - Lint the entire source and test tree with StandardJS (`npm run lint`) before running
    any tests, so formatting violations surface as a distinct failure
  - Execute the full Jest suite (`npm test -- --runInBand`) with the `MONGODB_URI`
    secret injected from GitHub repository secrets; when the secret is absent the Atlas
    tests are skipped automatically and the suite still passes
  - Upload the generated coverage report as an artifact after every run, regardless of
    test outcome, for inspection without re-running the suite
- **Atlas Secret Management:** The `MONGODB_URI` secret is stored in GitHub repository
  secrets and is never embedded in workflow files. The CI step notes in the job summary
  whether Atlas tests ran or were skipped, keeping the outcome transparent.

## Consequences

### Pros

- **Unified Toolchain:** Jest provides the test runner, assertion library, mock framework,
  and coverage reporter in a single dependency. This eliminates the need to coordinate
  separate tools such as Mocha, Chai, Sinon, and Istanbul, reducing version conflicts and
  onboarding overhead.
- **Zero-config for unit tests:** Pure function tests require no setup files, no
  lifecycle hooks, and no environment variables. The barrier to adding new tests for
  service and utility logic is minimal.
- **Realistic route testing without a test server framework:** Using `http.createServer`
  with native `fetch` means route tests exercise the real Express middleware pipeline,
  session handling, and error responses rather than a shallow stub. This catches
  integration issues that supertest-style shallow mounts would not.
- **Graceful Atlas gating:** The `test.skip` pattern keeps the full suite green in CI
  when the database secret is absent, allowing developers without Atlas access to run
  tests locally without failures.
- **Coverage as a first-class output:** Built-in coverage reporting with automatic CI
  artifact upload makes it straightforward to track untested paths across sprints without
  any additional tooling.
- **StandardJS enforced before tests:** Running the linter as a separate step before
  the test run means a style violation fails fast without burning test execution time.

### Cons

- **Jest startup overhead:** Jest's module isolation and transform pipeline adds
  measurable startup time compared to lighter runners such as Mocha or Node's built-in
  test runner. For a project of this size the overhead is acceptable, but it would grow
  if the test suite became very large.
- **`--runInBand` limits parallelism:** Serialising all test files prevents port
  conflicts between route test servers but forfeits Jest's default worker-pool
  parallelism. Tests that do not start servers could run faster in parallel, though the
  current suite size does not make this a bottleneck.
- **Atlas secret dependency for system tests:** Full end-to-end validation requires the
  `MONGODB_URI` secret to be present in CI. If the secret is rotated or removed, system
  tests silently skip rather than fail, which could mask regressions in Atlas-specific
  behaviour.
- **Manual mock maintenance:** Because the project uses the native MongoDB driver without
  an ODM, every route and model test must declare its own `jest.mock` setup. As the
  number of collections and helpers grows, keeping mock implementations consistent with
  real behaviour requires discipline.
- **No browser or end-to-end test coverage:** Jest tests the server layer only. UI
  rendering correctness and client-side JavaScript behaviour in EJS views and public
  scripts are not covered by the current strategy and would require a separate tool such
  as Playwright if full browser testing is added in a future sprint.

## Summary

**Jest 30** has been adopted as the unified test runner, assertion library, and mock
framework across all three testing tiers: pure unit tests for services and utilities,
integration tests for Express routes and model helpers using real HTTP servers and mocked
MongoDB collections, and system acceptance tests that connect to MongoDB Atlas to verify
seeded institution data and relationship lookups. The Atlas tests are gated on the
`MONGODB_URI` environment variable and skip gracefully when it is absent. **GitHub
Actions** enforces the full pipeline automatically on pull requests and pushes to `main`,
running StandardJS linting before the test suite and uploading the Jest coverage report
as an artifact on every build. This strategy gives the project a complete verification
layer from isolated business logic through to live database behaviour, without requiring
any tooling beyond Jest and the existing npm workflow.
