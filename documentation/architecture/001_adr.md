# ADR 001: Set Up Basic REST API with Express.js

## Context

We are building a web application for scheduling group consultations between students and
lecturers at Wits University. The backend must expose HTTP endpoints to support:

- User authentication (student and lecturer signup/login)
- Creating, joining, and cancelling consultation sessions
- Managing lecturer availability and constraints
- Serving a student and lecturer dashboard
- Logging all system actions

The application is required to use **Node.js** as the web server technology, per the
project's technical requirements. A lightweight, well-supported HTTP framework is needed
to structure route handling, middleware, and business logic in a maintainable way across
multiple sprints.

## Decision

We have decided to use **Express.js** as the HTTP framework for the Node.js backend, with
the following structure:

- **Framework:** Express.js for routing and middleware management
- **Request Parsing:** `express.json()` for incoming JSON payloads and
  `express.urlencoded()` for form submissions
- **Session / Auth Middleware:** `express-session` or JWT middleware (to be confirmed in
  a future ADR) for protecting lecturer and student routes
- **Folder Structure (MVC-style):**
  - `/src/routes/` — route definitions per resource (auth, consultations, availability, logs)
  - `/src/controllers/` — request/response handling and input validation
  - `/src/services/` — business logic and constraint enforcement (capacity, time overlap,
    availability windows, max consultations per day)
  - `/src/models/` — data schema definitions (to be populated once the database layer is
    established)
  - `/src/middleware/` — shared middleware (authentication guards, error handling, request
    logging)
- **Error Handling:** A centralised error-handling middleware will be registered to return
  consistent JSON error responses across all routes

An initial in-memory data store will be used during early development to unblock frontend
integration, and will be replaced with a persistent database in the following sprint (see
ADR 002).

## Consequences

### Pros

- **Minimal and Familiar:** Express is the standard choice for Node.js REST APIs with
  extensive documentation and community support, reducing onboarding time for team
  members.
- **Flexible Middleware Pipeline:** Middleware can be added incrementally (auth guards,
  logging, validation) without restructuring existing routes, which suits iterative sprint
  delivery.
- **MVC Separation:** The chosen folder structure cleanly separates routing, business logic,
  and data concerns, making code review and story assignment straightforward.
- **Constraint Enforcement Point:** The services layer provides a single, testable location
  to enforce all consultation constraints (no overlaps, capacity limits, availability windows,
  daily maximums), which is critical to core functionality.
- **Testable with Jest:** Controllers and services can be unit tested in isolation with Jest,
  supporting the project's CI/testing requirements from the first sprint onward.

### Cons

- **No Built-in ORM or Validation:** Input validation and data integrity must be implemented
  manually or via libraries such as `express-validator`. This adds developer effort.
- **Manual Security Hardening:** Express does not include built-in protection against common
  vulnerabilities. Libraries such as `helmet` and `express-rate-limit` will need to be added
  explicitly to meet the project's security requirements (password hashing, injection
  protection).
- **Boilerplate for Constraint Logic:** The constraint enforcement logic (e.g. overlapping
  consultation detection, availability checking) requires careful service-layer design to
  avoid bugs across sprints.

## Status

Accepted

## Summary

**Express.js** has been selected as the HTTP framework for the Consultation Scheduler
backend. Its simplicity, ecosystem support, and compatibility with Node.js make it the
appropriate choice given the project's technical constraints. The MVC folder structure
adopted here will provide a stable, reviewable foundation for all sprints, with the services
layer acting as the primary enforcement point for consultation business rules.
