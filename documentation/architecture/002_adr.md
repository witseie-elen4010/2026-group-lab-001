# ADR 002: Adopt MongoDB Atlas with the Official Node.js Driver

## Context

ADR 001 established Express.js as the backend framework and explicitly allowed an initial
temporary data store to be replaced by a persistent database in current sprint. The application now requires a real database layer to support:

- Persisting registered user accounts beyond process restarts
- Looking up users during authentication
- Storing institutional data for universities, faculties, and schools
- Modelling relationships between institutions using identifiers rather than only nested objects
- Supporting repeatable integration testing against realistic seeded data

At this point, the Express application structure has already been established, but the persistent database layer has not yet been implemented. The project needs a database solution that fits the Node.js and Express stack, supports incremental development, and can represent both user accounts and institutional reference data without requiring a large amount of upfront schema boilerplate.

The data layer will need to support at least the following domains:

- User authentication data, including usernames, emails, roles, and hashed passwords
- Institutional reference data for universities, faculties, and schools
- Parent-child relationships between institutions, for example a faculty belonging to a university and a school belonging to a faculty
- Lecturer Availability settings that constrain consultation scheduling
The database choice therefore needs to be formalised before the persistence layer is built.

## Decision

We have decided to use **MongoDB Atlas** as the persistent database for the project, and to access it using the **official MongoDB Node.js driver** rather than an ODM such as Mongoose.

The database layer will be introduced with the following structure:

- **Database Platform:** MongoDB Atlas
- **Driver:** `mongodb` Node.js driver
- **Connection Management:** A shared connection helper in `src/models/db.js`
- **Configuration:** `MONGODB_URI` for the connection string and
  `MONGODB_DB_NAME` for the logical database name, with `LetsTalk` as the planned default database name for development
- **Collections to be introduced initially:**
  - `User`
  - `University`
  - `Faculty`
  - `School`
  - `LecturerAvailability`
- **Data Access Pattern:** Collection-specific helper modules in `/src/models/`
  will encapsulate reads and writes, while routes will call those helpers after the database connection is established
- **Relationship Strategy:**
  - `Faculty.universityID` will store the parent university reference
  - `School.facultyID` will store the parent faculty reference
  - `School.universityID` will store the parent university reference
  - Model helpers will resolve these relationships using `ObjectId` conversion and collection lookups where needed
- **Authentication Storage:** User accounts will be stored in the `User` collection with a hashed password field (`passwordHash`), and authentication routes will read and write through dedicated model helpers rather than querying MongoDB directly from routes
- **Validation and Testing:** Atlas-backed integration tests will be added for collection availability, seeded records, relationship lookups, inserts, deletes, and constraint failures such as duplicate usernames

We are intentionally **not** introducing an ODM at this stage. The project is still small, and explicit model helpers with the native driver are sufficient for the current scope.

## Consequences

### Pros

- **Flexible document model:** MongoDB supports the planned mix of account data and institutional relationship data without requiring a rigid relational schema up front.
- **Natural fit for evolving stories:** The data model can start with a small set of core collections and expand as new stories introduce more consultation, scheduling, and institution-related data.
- **Low abstraction overhead:** Using the official driver keeps the persistence layer simple. Reads and writes are visible in plain JavaScript rather than hidden behind ODM conventions.
- **Clear separation of concerns:** A shared connection helper plus collection-specific model helpers will fit the folder structure already introduced in ADR 001.
- **Supports realistic testing:** Atlas-backed integration tests can later validate seeded institution data and database constraints against the same platform used in development.

### Cons

- **Manual schema discipline:** Without an ODM, validation rules and document shape expectations must be maintained either in MongoDB collection validators or in application code.
- **Relationship handling is manual:** Parent-child lookups across `University`, `Faculty`, and `School` will need to be implemented manually using `ObjectId` conversion and repeated collection queries unless a more abstract relationship layer is introduced later.
- **Schema evolution must be managed explicitly:** If early stories use placeholder values or simplified document shapes, the team will need a clear migration path as the model becomes richer.
- **Connection lifecycle must be managed carefully:** Routes and shared model helpers will depend on a common connection approach, so connection setup and teardown will need to be handled consistently as the application grows.
- **Atlas dependency for integration tests:** Full database tests require `MONGODB_URI` and seeded records to be present, which adds external setup to the test environment.

Accepted

## Summary

**MongoDB Atlas**, accessed through the **official MongoDB Node.js driver**, is the selected persistent database solution for the project. This decision replaces the temporary store anticipated in ADR 001 and establishes the planned persistence approach before the database layer is implemented. The project will use the native MongoDB driver, shared
connection management, and collection-level model helpers so that authentication data and institution reference data can be stored in MongoDB without introducing unnecessary framework overhead.
